From API to UI: Shipping Real-Time AI Features with NestJS + React (SSE)

Feb 20, 2025 • Achref Lajmi • Full-Stack / Real-Time Systems / Applied AI

What this post covers: how MASMEDIA AIDITOR ships real-time AI features end-to-end: a NestJS pipeline that processes content, a React dashboard that streams updates via SSE, and a review-first UX that keeps AI output controllable.

MASMEDIA AIDITOR was built to reduce the time between “a source appears” and “a publishable draft exists.” The goal wasn’t to “add AI”, but to ship a reliable product system: ingestion → processing → AI enrichment → real-time updates → human validation.

1) What the product does (in one sentence)

MASMEDIA AIDITOR turns raw incoming sources (links, feeds, media references) into structured items: normalized metadata, AI summaries/drafts, tags/categories, and a dashboard flow to review and publish.

2) Backend: the pipeline that makes “real-time” possible (NestJS)

The backend is designed around a pipeline mindset. Each item moves through stages, and each stage can report progress, store intermediate results, and fail gracefully without breaking the dashboard.

Ingestion

  • Accept a content request (URL, text, or media reference).
  • Validate, deduplicate, store the initial record.
  • Create a processing job and mark the item as queued.

Processing stages

A pragmatic multi-stage flow:

  1. Fetch / parse: retrieve content, extract usable text and metadata.
  2. Clean / normalize: remove noise, standardize fields, produce consistent structure.
  3. AI enrichment: summaries, key points, tagging/classification, optional rewrite.
  4. Persist + notify: store results and push live updates to the frontend.
Data model (simple & scalable):
  • ContentItem: source, status, timestamps, derived fields
  • ProcessingJob: stage, progress, errors, correlation id
  • Draft: summary, key points, suggested title, tags, category

Why SSE (not polling)

Polling makes the UI feel delayed and wastes requests. WebSockets are powerful but add complexity if you only need server → client updates. For this product, Server-Sent Events (SSE) were ideal: the server streams lightweight job events and the UI updates instantly.

Typical event types:

  • job.created — show the item immediately
  • job.progress — update stage + progress badge
  • job.completed — enable review actions
  • job.failed — show error + retry

3) Frontend: making the dashboard feel “alive” (React)

The UI isn’t just “showing data.” It’s the control surface for the pipeline. The key was designing a dashboard that stays responsive while jobs run in the background, and that communicates trust via clear states.

Streaming updates (SSE client)

  • Open an SSE stream when the dashboard loads.
  • Handle events incrementally (update a single item instead of re-fetching everything).
  • Recover from disconnects (retry/backoff + visible “reconnecting” indicator if needed).

State and UI patterns that feel production-ready

  • Optimistic UX: new item appears instantly with Processing….
  • Explicit states: queued → extracting → summarizing → ready.
  • Failure states: “failed” with a clear reason + retry action.
  • Fast rendering: update by id; avoid full list refresh.
  • Quality UX: skeleton loaders, empty states, filters (status/category).
Recruiter signal: Real-time systems are not only backend engineering. The user trusts the product when the UI explains what is happening, what failed, and what they can do next.

4) The AI layer: useful, controllable, and review-first

The AI layer is a pipeline stage — not the entire product. That design choice is important because AI calls can fail, vary in quality, and should never block the user from progressing.

What AI generates

  • Summary in an editor-friendly structure
  • Key points for quick review
  • Tags / category to support filtering and routing
  • Optional rewrite to match a tone or format

Guardrails that keep quality high

  • Review-first UI: AI output is always validated by a human in the dashboard.
  • Traceability: show the source next to the generated draft.
  • Resilience: if AI fails, mark the item failed and allow retry.
  • Cache intermediate results: avoid regenerating when not needed.

5) Lessons learned (what I’d repeat)

  • Real-time UX is a product feature: progress events increase trust and reduce confusion.
  • SSE is the simplest real-time tool when you only need server → client updates.
  • Pipelines must be resumable: treat external calls (including AI) as unreliable.
  • Small events beat big payloads: push what changed, keep the UI fast.
  • Quality comes from workflow: AI becomes valuable when it’s editable, reviewable, and traceable.

6) What I’d build next (scaling direction)

  • Add a job queue (e.g., BullMQ + Redis) for higher load and better scheduling.
  • Introduce correlation ids and tracing to follow one item across stages.
  • Improve retry/backoff and add a dead-letter strategy for repeated failures.
  • Rate-limit and cache AI operations to control cost and latency.

Tech Stack

NestJS • React • MongoDB • SSE • RxJS • Docker • LLM APIs / OpenAI • FFmpeg (when media processing applies)
